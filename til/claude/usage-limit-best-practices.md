# Claude 사용 한도 모범 사례(Usage Limit Best Practices)

> [!tldr] 메시지 수가 아니라 컨텍스트 크기와 캐싱 전략이 사용량을 결정한다

## 핵심 내용

### 사용량에 영향을 미치는 요소

단순히 "메시지 수"가 아닌 다음 모든 요소가 사용량을 소비한다:

- 메시지 길이
- 파일 첨부 크기
- **현재 대화 길이** — 대화가 길어질수록 매 메시지마다 전체 컨텍스트를 처리
- 도구 사용 (웹 검색, 리서치 등)
- 모델 선택
- 아티팩트 생성

### 8가지 모범 사례

**1. 대화 계획하기**
전송 전에 필요한 것을 명확히 정의. 여러 관련 질문을 하나로 통합해 왕복 횟수를 줄인다.

**2. 구체적이고 간결하게**
모호한 질문은 후속 메시지를 유발한다. 관련 맥락을 처음부터 포함시킨다.

**3. 채팅 검색/메모리 활용**
이전 대화 내용을 반복 설명하지 말고 참조. 유료 플랜은 이전 대화 검색 가능.

**4. 유사 요청 통합**
질문 5개를 각각 보내지 말고 한 메시지에 묶기. 메시지 수를 직접 줄이는 가장 단순한 방법.

**5. 전송 전 검토**
처음부터 완성도 높게 작성해 후속 질문을 최소화한다.

**6. Project 지식 베이스 캐싱** ← 가장 효과적
프로젝트에 업로드한 문서는 캐시됨. 반복 참조해도 매번 업로드하는 것보다 사용량이 적게 차감된다. RAG 모드로 용량 확장 가능.

**7. 설정 > 사용에서 모니터링**
- **현재 세션**: 5시간 세션 한도 중 사용량 및 남은 시간
- **주간 한도**: Opus 또는 전체 모델의 한도 리셋 시점

**8. 캐싱 극대화**
자주 참조하는 문서는 Project 지식 섹션에 업로드. 같은 콘텐츠를 많이 쓸수록 캐싱 이득이 커진다.

### 가장 효과적인 것부터

1. **프로젝트 캐싱** — 행동 변화가 아닌 구조적 절감. 한 번 설정하면 지속 효과
2. **대화 길이 관리** — 긴 대화를 새로 시작하는 것만으로도 크게 절감
3. **유사 요청 통합** — 단순하지만 즉각적 효과

## 예시

**사용 사례별 요령**

| 사용 사례 | 요령 |
|----------|------|
| **코딩** | 초기 메시지에 환경/코드 전체 맥락 제시 |
| **글쓰기** | 요구사항·독자·포인트를 처음부터 개요. 텍스트 전체를 한 번에 |
| **연구/분석** | 연구 질문과 데이터를 하나의 구조화된 메시지로 |

**비효율적 vs 효율적**
```
# 비효율적
메시지 1: "파이썬 코드 봐줘"
메시지 2: "아, 파일 읽는 코드야"
메시지 3: "에러는 이거야..."

# 효율적
메시지 1: "파이썬 파일 읽는 코드인데 FileNotFoundError 발생함. [코드 전체] [에러 전체]"
```
